# 第3章：Gitで「セーブポイント」を作る

## Gitとは

**Git**は、ファイルの変更履歴を記録するシステムです。

ゲームのセーブに例えると分かりやすいです：

| ゲーム | Git |
|--------|-----|
| セーブする | **コミット**する |
| セーブデータに名前をつける | **コミットメッセージ**を書く |
| セーブを読み込んで戻る | 過去のコミットに**戻る** |
| クラウドにセーブを同期 | **プッシュ**する |
| クラウドからセーブを取得 | **プル**する |

## なぜGitが必要か

Gitがないと、こうなります：

```
企画書_v1.docx
企画書_v2.docx
企画書_v2_最終.docx
企画書_v2_最終_修正.docx
企画書_v2_最終_修正_本当に最終.docx  ← どれが最新？
```

Gitがあると：

```
企画書.docx  ← 常に1つ。履歴はGitが管理。
```

## 覚える操作は3つだけ

### 1. コミット（保存する）

「この時点の状態を記録する」操作です。

**VS Codeでの手順：**
1. 左サイドバーの「ソース管理」アイコン（分岐マーク）をクリック
2. 変更されたファイルが一覧で表示される
3. 「+」ボタンでファイルをステージ（保存対象に追加）
4. メッセージ欄に**何を変更したか**を書く
5. 「コミット」ボタンを押す

**コミットメッセージの書き方：**
```
良い例：「ダッシュボードに今月の売上グラフを追加」
悪い例：「修正」「更新」「変更」
```

何を変えたか、未来の自分が読んで分かるように書きましょう。

### 2. プッシュ（GitHubに送る）

コミットした内容を**GitHubにバックアップ**する操作です。

**VS Codeでの手順：**
1. コミット後、「変更の同期」ボタン（↑↓マーク）をクリック

これだけです。パソコンが壊れても、GitHubにデータが残っています。

### 3. プル（GitHubから取得する）

別のPCで作業した内容を**このPCに取り込む**操作です。

**VS Codeでの手順：**
1. 「変更の同期」ボタン（↑↓マーク）をクリック

プッシュと同じボタンです。VS Codeが自動で「送るもの」と「受け取るもの」を判断してくれます。

## 日常の流れ

```
朝（作業開始）
  → VS Codeを開く
  → 「変更の同期」を押す（前日の作業を取得）

日中（作業中）
  → ファイルを編集
  → キリのいいところでコミット
  → メッセージを書いて保存

夕方（作業終了）
  → 最後のコミット
  → 「変更の同期」でプッシュ
```

## 2台のPCで同じ環境にする

GitHubを経由すれば、**デスクトップPCとノートPCで同じプロジェクトを作業**できます。

```
デスクトップPC → プッシュ → GitHub → プル → ノートPC
ノートPC → プッシュ → GitHub → プル → デスクトップPC
```

### 初回セットアップ（2台目のPC）

1. GitHub Desktopまたは VS Codeで「Gitリポジトリのクローン」
2. GitHubのURLを入力
3. 保存先フォルダを選択

以降は「変更の同期」ボタンだけで、どちらのPCでも最新の状態で作業できます。

## よくある失敗と対処

| 失敗 | 対処 |
|------|------|
| コミットし忘れて帰った | 翌日コミットすればOK。データは消えていない |
| プッシュし忘れて別PCで作業 | 両方コミットして、マージ（統合）する |
| 間違ったファイルをコミットした | `git revert` で取り消せる（Claude Codeに頼もう） |
