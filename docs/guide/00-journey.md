## Day 14：転換点（4月3日）

この日、すべてが変わった。

### 課題の認識

- GitHub Desktopの英語インターフェースに苦戦していた
- Git特有の概念（コミット、プッシュ、ブランチ）が掴めなかった
- 1ヶ月分の成果がバックアップされていない不安があった
- Claudeデスクトップアプリ単体での作業に限界を感じていた

### 決断：VS Codeへの移行

Claudeデスクトップアプリ単体から、**VS Code + Claude Code + Git** の統合環境へ移行することを決めた。

### 最初の作業：Git環境の構築

まず既存の4プロジェクトの状態を確認した：

```
eo-learning   → Git初期化済み、GitHub接続済み
keiei-apps    → Git初期化済み、GitHub接続済み
lifeline      → Git初期化済み、GitHub接続済み
lifelog        → Git初期化済み、だがGitHubと大幅にズレ
```

4つともGitとGitHub接続は済んでいたが、**lifelog（DayMatrix）だけが同期が取れていなかった。** ローカルに3件の未Push、GitHubに119件の新しいコミット。ノートPCで作業した分がデスクトップPCに取り込まれていなかった。

### 同期作業で起きたこと

lifelog の同期は一筋縄ではいかなかった：

1. **鍵ファイルの衝突** — APIキーなどの機密ファイルがrebaseを阻んだ。一時退避して対処
2. **.gitignoreの競合** — リモートとローカルで異なる内容。両方を統合して解決
3. **鍵ファイルの復元** — 一時退避先が消えていた。Gitの履歴から復元に成功

この作業だけで1時間以上かかった。しかし結果として、**1ヶ月分の全データがGitHubに安全にバックアップされた。**

### ブランチの統一

全プロジェクトのブランチ名を `master` から `main` に統一した。これは業界標準に合わせるためと、全プロジェクトで統一感を持たせるため。

### 未コミット変更のコミット

各プロジェクトに溜まっていた未コミットの変更を、意味のある単位でコミットした：

- **eo-learning**: HTMLの修正と写真追加
- **keiei-apps**: 家賃入金一覧表データ（8ファイル）
- **lifelog**: v0.18の28ファイル（新機能丸ごと）を2つのコミットに分割
- **lifeline**: 変更なし（クリーン）

## 新しいプロジェクトの誕生

### ContentOS → Hibana（SNS管理アプリ）

Facebook投稿の管理がNotionと手動確認に散在していた問題を解決するため、新しいアプリを作った。

**技術選定の議論：**
- Next.js + SQLite（ローカル完結）を選択
- Supabaseは不要と判断（SNS管理はデスクトップ作業が中心）
- Dropbox同期だけで2台のPC間で共有できる

**Phase 1で作ったもの：**
- ダッシュボード（今月の統計・次の予定・最近の投稿）
- 投稿カレンダー（月表示・日付に記事を色分け表示）
- 記事ストック（CRUD・タグ・ステータス管理）
- インサイト入力（閲覧数・反応・予測 vs 実績比較）

`npx create-next-app` からビルド成功まで約30分。UIの作成を含めて1時間半。

### EO Survey（EO統合サーベイ）

Notionにあった3つのページ（開発手順書・フォーラムサーベイ・モデレーターサーベイ）から、**要件定義書を自動生成**した。

モデレーターサーベイ（32問）とフォーラムサーベイ（29問）の全設問仕様、スキップロジック、技術構成、30期の問題と対策、年次運用フローまでを1つのSPEC.mdにまとめた。

## Playbookの誕生

### きっかけ

1ヶ月で20人以上のEO仲間とワンオンワンをしてきた。弁護士、医師、EC事業者、PR会社、太陽光発電、知育玩具、スペースシェアリング——業種はバラバラだが、全員に共通していたのは「AIで何かを変えたい」という想い。

それぞれにカスタムプレゼンを作り、個別に教えてきた。しかしこのやり方にはスケールの限界がある。

### 構想

「ワンオンワンのノウハウを体系化し、コミュニティでブラッシュアップできるドキュメント」を作ることにした。

**VitePress**（Markdownで書くだけでドキュメントサイトになる）を選択。VS Codeで書いてGit管理する——自分が教えている方法そのもので作る。

### 構成

入門編として6章 + 付録を執筆：

1. **なぜ自分で作るのか** — Power BIの限界、2週間で8プロジェクトの実例
2. **VS Codeを司令塔にする** — インストール、日本語化、画面の見方
3. **Gitでセーブポイントを作る** — コミット＝ゲームのセーブ、3つの操作だけ
4. **Claude Codeと開発する** — 指示の出し方、CLAUDE.md、スキル
5. **最初のアプリを作る** — 1時間で動くWebアプリの実践
6. **プロジェクトを管理する** — ワークスペース、バージョン管理、日常ルーティン
7. **付録** — トラブルシューティング集

### 次のステップ

実践編として以下を追加予定：
- **DayMatrix編** — v0.1→v0.19の進化過程（アーキテクチャと設計思想のみ公開）
- **MoneyGrid編** — 会計システムを自分で管理する方法
- **ワンオンワンから見えたパターン** — 20人の経営者に共通する課題と解決策
- **コミュニティを作る** — Facebookグループでの継続的な学びの場

## 学んだこと

### 技術的なこと

- **コミットはゲームのセーブ。** この比喩で全てが腑に落ちた
- **CLAUDE.mdが最重要ファイル。** これがあるとAIが毎回文脈を理解してくれる
- **VS Codeワークスペースで一覧管理。** プロジェクトが増えても迷わない
- **Git同期は朝と夕方の習慣。** 2台のPCで常に最新を保つ

### 仕事の進め方

- **完璧を目指さず、まず動くものを作る。** HibanaはPhase 1を1.5時間で完成させた
- **既存資産を活かす。** Notionの情報から要件定義書を自動生成した
- **統合と整理を恐れない。** 8プロジェクト→7プロジェクトへの統合は正しい判断だった
- **管理のための管理はしない。** `/project-status` の1コマンドで十分

### 教えることについて

- **個別プレゼンは火をつける。** 20人全員が「やりたい」と言った
- **しかしスケールしない。** だからPlaybookとコミュニティが必要
- **最高の教材は自分の実体験。** DayMatrixの進化過程そのものが教材になる
- **公開範囲は慎重に。** 個人データを含むプロジェクトはアーキテクチャのみ共有
