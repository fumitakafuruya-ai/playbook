# 第6章：プロジェクトを管理する

## プロジェクトが増える問題

アプリ作りが楽しくなると、プロジェクトがどんどん増えます。

筆者の場合、2週間で8プロジェクトになりました：

| # | プロジェクト | バージョン | 内容 |
|---|------------|-----------|------|
| 1 | 業務管理アプリ | v0.19 | 日常業務の統合管理 |
| 2 | Hibana | v0.1 | SNS管理 + プレゼン |
| 3 | 自社管理アプリ | v0.1 | 自社経営管理 |
| 4 | EO Survey | v0.1 | 組織サーベイ |
| 5 | ライフライン | v1.0 | 人生年表 |
| 6 | 5% Update | v1.0 | 月次振り返り分析 |
| 7 | 取引先分析 | v0.1 | 取引先の売上・商品分析 |
| 8 | Playbook | v0.1 | この文書（ナレッジベース） |

管理しないと「あのファイルどこだっけ？」「どっちのPCが最新？」が頻発します。

## 管理の3本柱

### 1. フォルダ構造を統一する

全プロジェクトを1つの親フォルダに置きます：

```
D:\dev\
├── apogrp/
├── eo-survey/
├── lifeline/
├── my-app/
├── sns-manager/
├── 5percent-update/
├── playbook/
└── all-projects.code-workspace  ← これで全部開く
```

### 2. CLAUDE.mdを全プロジェクトに置く

各プロジェクトのルートに `CLAUDE.md` を置きます：

```markdown
# プロジェクト名
> バージョン: v0.1 | 開始: 2026-04-03

## 概要
（1-2行で説明）

## 技術スタック
（使っているツール）

## 起動方法
（コマンド）
```

これがあると：
- 新しいClaude Codeセッションでも即座に文脈を理解してくれる
- 半年後の自分が見ても何のプロジェクトか分かる
- 他の人に引き継ぐときも説明が楽

### 3. ステータスを一発確認

Claude Codeのスキルで `/project-status` を登録すると：

```
/project-status
```

一発で全プロジェクトの状態が分かります：

| プロジェクト | GitHub同期 | 未コミット | 最終更新 |
|------------|-----------|-----------|---------|
| 業務管理アプリ | 最新 | 0件 | 4/7 — v0.19 |
| Hibana | 最新 | 0件 | 4/7 — プレゼン統合 |
| 自社管理アプリ | 最新 | 0件 | 4/3 — リネーム |

**「要対応」があれば末尾に表示** されるので、漏れがなくなります。

## バージョン管理のルール

| バージョン | 意味 |
|-----------|------|
| v0.1 | 作り始めた段階 |
| v0.x | 機能を追加するたびに上がる |
| v1.0 | 本番運用開始 |
| v1.x | 本番運用中の改善 |

CLAUDE.mdのバージョンを更新するだけでOK。大げさな管理は不要です。

## 日常のルーティン

### 朝（作業開始）
1. VS Codeでワークスペースを開く
2. `/project-status` で全体を確認
3. 今日やるプロジェクトを決める

### 作業中
1. 変更したらこまめにコミット
2. コミットメッセージに「何を変えたか」を書く
3. 区切りがついたらプッシュ

### 夕方（作業終了）
1. 全プロジェクトをプッシュ
2. `/project-status` で未コミットが0件か確認

## プロジェクトの統合・分割

プロジェクトが増えすぎたら、統合を検討します。

**統合の判断基準：**
- 同じテーマのプロジェクトが2つある → 統合
- 1つのプロジェクトが大きすぎる → 分割

筆者の例：EO Learning（プレゼン資料）をHibana（SNS管理）に統合しました。プレゼン資料はSNS活動の一部だったためです。

## まとめ

| やること | ツール |
|---------|-------|
| 全プロジェクトを一覧で見る | VS Code ワークスペース |
| 各プロジェクトの説明を書く | CLAUDE.md |
| 全体の状態を一発確認 | `/project-status` スキル |
| バージョンを記録する | CLAUDE.mdに記載 |
| 毎日の同期を忘れない | 朝と夕方の習慣化 |

これで、プロジェクトが10個、20個に増えても管理できます。
